[문제 요약]

- 등산 게임은 일차원으로 이루어진 지도 위에서 이루어진다
- 지도에는 산들이 연속으로 나열되어 있으며, 각각의 산은 높이를 가진다
- 또한 아래와 같은 조건들을 가지며 게임이 진행된다

```
1. 등산가의 이동 조건

- 매 등산 시뮬레이션마다 등산을 통한 산 사이 이동은 현재 위치보다 오른쪽에 위치한 산으로만 이동할 수 있다
- 등산을 통한 산과 산 사이 이동에 있어서 항상 현재 산의 높이보다 더 높은 산으로만 이동이 가능

2. 케이블 카

- 케이블 카는 특정 산에서만 탈 수 있음
- 케이블 카를 타면 현재 위치를 포함한 임의의 산으로 이동할 수 있다
	- 이때 산의 높낮이는 상관x
- 케이블 카를 탄 이후에도 등산을 이어가며 이 또한 오른쪽에 위치한 더 높은 산으로만 이동 가능

3. 등산 시뮬레이션

1) 등산가는 시뮬레이션을 시작할 산을 자유롭게 선택 가능
2) 등산가가 현재 위치보다 오른쪽에 위치한 산으로 오르막 이동에 성공할 때마다 1,000,000점을 얻는다
3) 만약 등산가가 케이블 카를 탈 수 있는 산에 도착하면 케이블 카를 이용
	- 이때 케이블 카 이용에 성공하면 1,000,000점을 얻음
4) 케이블 카 이용에 성공했다면, 이 후 다시 등산을 시작
	- 이 또한 성공할 때마다 1,000,000점을 얻음
5) 최종적으로 위치한 산의 높이만큼 점수를 얻는다
- 등산 시뮬레이션 결과 얻을 수 있는 총 점수는 2, 5점수의 총합
```

<등산 시뮬레이션>
- n개의 수열에서 증가하는 부분 수열 중 가장 긴 수열을 찾는 문제로 치환할 수 있다
- 단, 케이블 카를 탈 경우 높이에 상관없이 이동이 가능하다는 제약사항이 존재한다
- 케이블 카를 고려하지 않고 문제를 접근해보자
	- n개의 수열에서 증가하는 최장 부분 수열을 찾는 문제다
	- 이는 이분 탐색을 이용하면 해결할 수 있다
	- O(NlogN) 복잡도 <- 시간초과가 발생할 수 있으나, 일단 시도해봐야 함
- 케이블 카에 의한 효과는 어떻게 반영해야 할까?
	- 케이블 카가 도움이 되기 위해서는 케이블 카 없이 등산하는 것보다 길어야 한다
	- 길이가 같다면, 마지막 도착 산의 높이가 더 높아야 한다
	- 그런데 생각해보면, 항상 케이블 카를 타는게 이득이다
	- 케이블 카까지 등산은 한 번이상 수행할 것이고, 케이블 카를 타고 처음으로 돌아가서 등산을 수행하자
	- 이럴 경우 항상 케이블 카를 타는 것이 최대이거나 같다
	- 따라서 0번 산부터 m번 산까지 부분 수열의 LIS를 구하고, 다시 처음부터 끝까지 LIS를 구하면 된다

	- 단, m번째 산을 반드시 방문해야 한다
	- m번째 산에서 왼쪽으로 감소하는 가장 긴 부분 수열을 구하자
	- 그리고 그 길이를 기록한다
	- 0번 인덱스부터 끝까지 LIS를 구한다

	- 단, LIS를 만족하는 길이가 여러 개라면, 마지막 아이템이 가장 큰 값인 수열이어야 한다

가장 긴 증가하는 부분수열을 응용해서 가장 긴 감소하는 부분수열을 구할 수 있다

그렇지 않으면, lisList에서 적당한 위치를 찾는다
적당한 위치라는 것은 오른쪽에서 왼쪽으로 배열을 봤을 때 lower_bound를 찾으면 된다

[최적화]

- 이분 탐색 구현은 시간 초과가 발생함
- 등산 시뮬레이션을 O(logN)만에 수행해야 함
- 등산 시뮬레이션은 항상 [1,m]구간까지의 최댓값과 [1,끝] 구간까지의 최댓값을 구해야 함
- Segment Tree를 구성하면 구간 최댓값을 O(logN)만에 얻을 수 있음
- 단, 구간 정보를 build하고 update할 때 제약사항이 존재함
- 왼쪽 구간의 정보와 오른쪽 구간의 정보를 합칠 때, 기존의 정보로 업데이트할 수 없음?
- 왼쪽 구간 정보가 있더라도 오른쪽 구간 정보랑 합칠 때는 새로 계산해야 하지 않나?
- 왜냐하면 왼쪽 구간과 오른쪽 구간이 합쳐지는 순간 새로운 최장 길이 수열이 발생할 수 있기 때문 -> 별개의 최댓값 중 더 큰 값을 선택하는 것은 요구사항에 어긋남
- 그럼 어떻게 설계해야 하나?

[최적화2]

- Segment Tree를 변형해서 사용해야 함
- Segment Tree의 인덱스는 높이로 설정하고, 값은 해당 높이까지의 최대 길이를 저장하도록
- 단, Segment Tree는 최댓값을 얻기위한 수단이며, 실제로는 dp에서 최댓값을 확인해야 함
- 왜냐하면 똑같은 높이가 또 추가되면 Segment Tree의 값이 오염되기 때문

[최적화2에 따른 설계]

<자료구조>

- 산 배열
	- idx는 순서
	- arr[idx] = idx 번째 산의 높이

- dp 배열
	- idx는 순서
	- dp[idx] => 1~idx까지 LIS
	- 업데이트 시점과 무관

- 원본 배열
	- idx는 높이(height) idx=h
	- 1부터 h까지 최대 길이값을 저장하는 배열
	- ori[]

- Segment Tree
	- ori값을 기반으로 노드별 값을 가진 트리
	- Complete Binary Tree
	- seg[]

- HashMap or TreeMap
	- 인덱스(height) -> stack(vector)
	- stack에는 변경 이력을 저장

<알고리즘>

1. 빅뱅(100)

- 처음 n개의 산에 대한 dp와 seg를 구축해야 함
- n번 반복하면서, n번의 Query와 n번의 Update를 순서대로 수행
- 처음 ori 배열은 모두 0으로 채워져 있음
- 첫 번째 산에 대해 Query(1, h-1)을 수행
- dp[1] = Query(1, h-1) + 1
- map[h].push_back(value)
- Update(i, dp[i])
- 두 번째 산부터 n번째 산까지 마찬가지로 수행

2. Query(1, h)

- 1~h까지의 구간에서 최댓값을 가져옴
- seg에서 tree traversal을 수행하며 단순히 해당 구간에서 최댓값을 가져오는 함수
- 리턴은 최댓값과 그 때의 h를 같이 리턴하는 걸로 하자

3. Update(h, value)

- ori[h] = value
- seg 업데이트
- 부모로 이동하면서, 부모 노드를 업데이트

4. 우공이산(200)

- arr의 마지막에 h를 추가
- dp[i] = Query(1, h-1) + 1
- map[h].push_back(value)
- Update(h, dp[i])

5. 지진(300)

- 산 배열 길이 -1
- map[마지막 산의 높이].pop_back()
- dp는 상관x
- Update(마지막 산의 높이, map[마지막 산의 높이].back())
